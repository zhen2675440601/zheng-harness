- Task 1 bootstrap established the MVP Go package roots (`cmd/agent`, `internal/{domain,runtime,tools,verify,store,memory,config,llm}`) with package-level ownership docs to keep future implementation aligned with the planned architecture.
- Task 2 uses explicit domain structs plus narrow ports (`Model`, `ToolExecutor`, `MemoryStore`, `SessionStore`, `Verifier`) so the runtime can be tested with deterministic fakes and without any infrastructure coupling.
- Task 5 establishes a minimal verification baseline through a single GitHub Actions workflow plus Makefile targets (`fmt`, `lint`, `test`, `test-race`, `test-cover`) so later tasks can share one command vocabulary locally and in CI.
- Task 6 keeps provider wiring outside domain/runtime by using a small `internal/llm` provider interface, env/flag-backed `internal/config` loading, and versioned prompts under a single `internal/config/prompts` path.
- Task 3 evolves runtime into a bounded plan-execute-verify loop with explicit terminal statuses, replanning on failed verification, and fake-driven tests for success, retry exhaustion, budget exhaustion, interruption, and fatal errors.
- Task 4 uses a registry + safety-policy split: definitions declare schema/timeout/risk metadata while execution adapters stay focused on workspace-scoped file/search/shell operations enforced by allowlists and path boundaries.
- Task 7 keeps verification evidence-driven by separating reusable check types from policy classification, allowing contradiction/insufficient-evidence outcomes to return bounded correction instructions instead of silent pass/fail decisions.
- Task 8 persists sessions/plans/steps as typed JSON payloads inside SQLite tables while keeping `internal/domain` unchanged, and constrains durable memory through explicit `scope/type/source/confidence/provenance` validation plus session-scope read isolation and global-scope write rejection.
- Task 9 CLI keeps all interface wiring in `cmd/agent` by using standard-library `flag.FlagSet` subcommands, deterministic `FakeModel`/`FakeToolExecutor`/`FakeVerifier` stubs, and a thin session-ID alias store so persisted session IDs stay human-facing while `runtime.Engine` keeps its existing `task.ID + "-session"` behavior.
- Task 9 follow-up found that `runtime.Engine` persists sessions and step history under `task.ID + "-session"`; CLI tests only pass when `cmd/agent` creates the user-facing row up front and rewrites all runtime session/step writes back to that same external session ID.
- Task 9 interrupt follow-up showed that even with the same final session ID, it is safer to route the initial placeholder `SaveSession` through the same alias-store adapter instance used by `runtime.Engine`, keeping the write path consistent when SQLite resolves `ON CONFLICT(id)` updates.
- A second Task 9 interrupt follow-up showed that post-`Run` status normalization in `cmd/agent` must re-save the session if it changes `running/empty` into `interrupted` or `fatal_error`; otherwise CLI output and persisted SQLite status can diverge.
- `SQLiteSessionStore.SaveSession` upserts both `status` and `task_id`, so any CLI-layer terminal-status rewrite must supply a non-empty `TaskID`; otherwise later `ResumeSession` calls fail when they try to load the plan by `task_id`.
- Task 10 regression coverage can stay deterministic by loading small JSON replay fixtures from `testdata/runtime/` and driving the existing runtime fakes from fixture plans/actions/observations/verification results instead of introducing any new domain contracts or external harness.
- Task 10 unsafe-tool regression is easiest to keep honest by wrapping the real `tools.NewExecutor` inside the existing fake tool executor hook, so runtime tests still use fake adapters by default while the safety-policy path is exercised end-to-end for allowlist rejection.
