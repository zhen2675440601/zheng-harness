- Task 1 bootstrap established the MVP Go package roots (`cmd/agent`, `internal/{domain,runtime,tools,verify,store,memory,config,llm}`) with package-level ownership docs to keep future implementation aligned with the planned architecture.
- Task 2 uses explicit domain structs plus narrow ports (`Model`, `ToolExecutor`, `MemoryStore`, `SessionStore`, `Verifier`) so the runtime can be tested with deterministic fakes and without any infrastructure coupling.
- Task 5 establishes a minimal verification baseline through a single GitHub Actions workflow plus Makefile targets (`fmt`, `lint`, `test`, `test-race`, `test-cover`) so later tasks can share one command vocabulary locally and in CI.
- Task 6 keeps provider wiring outside domain/runtime by using a small `internal/llm` provider interface, env/flag-backed `internal/config` loading, and versioned prompts under a single `internal/config/prompts` path.
- Task 3 evolves runtime into a bounded plan-execute-verify loop with explicit terminal statuses, replanning on failed verification, and fake-driven tests for success, retry exhaustion, budget exhaustion, interruption, and fatal errors.
- Task 4 uses a registry + safety-policy split: definitions declare schema/timeout/risk metadata while execution adapters stay focused on workspace-scoped file/search/shell operations enforced by allowlists and path boundaries.
- Task 7 keeps verification evidence-driven by separating reusable check types from policy classification, allowing contradiction/insufficient-evidence outcomes to return bounded correction instructions instead of silent pass/fail decisions.
- Task 8 persists inspectable state with explicit SQLite tables for sessions, plans, steps, artifacts, and memory entries, while memory writes remain policy-gated by scope/type/source/confidence validation rather than opaque blobs or vector search.
- Task 9 can stay minimal by wiring the existing runtime/store/memory/tools/verify packages behind `run`, `resume`, and `inspect` subcommands with optional JSON output, while a stub model provides a CLI contract placeholder until real model integration arrives.
- Import-cycle remediation worked best by making `internal/domain/memory.go` the single source of truth for memory contracts and converting `internal/memory` into validation/policy logic over domain-owned types instead of duplicating structs.
- For cross-package memory reuse, a shared `domain.MemoryEntry` with richer fields (`SessionID`, `ProjectKey`, `Key`, `Content`, `Provenance`, `UpdatedAt`) reduced adapter drift across runtime, prompts, and SQLite-backed stores.
- CLI drift was resolved by treating `cmd/agent/cli.go` as the authoritative entrypoint and reducing `cmd/agent/main.go` to a thin `main()` wrapper that just calls the modern `runCLI(context, args, stdout, stderr)`.
- Additive `VerificationResult.Status` fields should be normalized explicitly in replay/persistence tests; legacy expectations that omit status can fail deep-equality checks even when behavior is otherwise correct.
- Final verification fixes showed that shared memory rules must flow through one policy entrypoint (`memory.ValidateEntry`) so store adapters cannot silently diverge on allowed scopes, required fields, or global-write restrictions.
- Root CLI help stays more robust when `runCLI` treats empty args and root `-h`/`--help`/`help` as usage requests before subcommand dispatch, instead of relying on subcommand-local flag parsing.
